1. What ARA is

An ARA (Agent-Native Research Artifact) is a per-project repository that serves as the research memory, plus three skills that operate on it. There is no orchestrator: the host agent does the work, and coordination happens entirely through the files. One agent, one thread; the recorder blocks the loop briefly while it writes, then work resumes. The Seal is the only asynchronous element, a separate periodic session.

Closure moment
Any point where something resolves: an experiment produces a result, a hypothesis is confirmed or killed, a dead end is hit, or the session ends. Most closures happen mid session. Raw evidence lands in /evidence continuously; only the epistemic layers wait for outcomes.
Foresight
Read only, pulled, never scheduled. Answers questions whose answers are not literally recorded by composing existing claims, labeling grounded steps versus speculative leaps, attaching confidence, and naming what observation would falsify the prediction.
Repository
Four layers. /logic holds claims, concepts, and heuristics as structured entries with IDs. /trace holds the exploration graph including dead ends. /evidence holds raw outputs. /src holds code and configs with rationale. Cross references by ID make the whole thing a traversable knowledge graph of how the project went.
Claim schema
Every crystallized entry in /logic carries: Statement · Conditions · Sources · Status · Confidence · Provenance · Falsification
Example: C-0142 cites T-0891 (trace) → E-3304 (evidence).
Refuted claims keep their record with Status flipped, never deleted.
Seal
Periodic audit in three levels: structural integrity, epistemic rigor, execution reproducibility. Failure blocks nothing; findings are written back as work items and the artifact carries no trust stamp until they are addressed.

The research-manager, zoomed in

Every step is the same LLM following a rubric in the skill file. The router decides what type of object each harvested event is; only belief bound events pass through a ripeness gate.

Harvester and Router
The harvester scans everything since the last checkpoint and extracts discrete events; the router types each one. Beliefs to /logic (via the gate), attempts to /trace, raw outputs to /evidence, code and configs to /src. Every step is the same LLM following a rubric in the skill file, not a separate service.
Maturity gate
Applies only to belief candidates. Ripe means: resolved outcome, linked evidence, precise enough to carry a falsification clause, and generalizes beyond the single run. Anything failing waits in staging and is re-checked at later closures, where it matures, merges into an existing claim, or dies.
Claim record
A crystallized claim carries Statement, Conditions, Sources, Status, Confidence, Provenance, and Falsification. Corrections are new entries citing the old claim by ID; refuted claims keep their record with Status flipped, never deleted.

How are artifacts updated?

Once the manager has written to a layer, the layers do not all behave the same way, and the differences are load bearing rather than incidental. The governing rule: anything another record can cite as a source must be either immutable or version pinned, so that a citation never silently goes stale. This is exactly what the Seal's structural check (do cited IDs still resolve) depends on.

Layer Mutability How it changes
/evidence Write once, immutable Append only. A rerun with different numbers is a new entry with a new ID, never an overwrite, because claims cite evidence by ID and mutating it would invalidate them.
/trace Append only body, mutable status Nodes (hypothesis, intervention, observation) are frozen once written; only a small lifecycle marker moves, e.g. exploring becomes superseded by node 47.
/src Freely mutable Code and configs are edited normally; git carries their history, not the artifact. Citations into src must therefore pin a version (commit or inlined parameter values), not just name a file.
/logic Append only records, mutable status New claims are appended with fresh IDs. Existing claims change only by status transition (supported to refuted, scope widened or narrowed); the old record is preserved and the correction cites it by ID. Records are never deleted.
staging Freely mutable, never cited The holding area for belief candidates that are not yet ripe. Entries are revised and deleted freely as beliefs form and collapse. Precisely because it is volatile, nothing may cite staging as a source; a candidate becomes citable only once it crystallizes into /logic.

Evidence and trace bodies never move, so citations into them stay valid forever; src moves, so citations into it pin a version; logic grows monotonically as an audit trail of belief, never a silently rewritten current state; staging is the deliberate exception that absorbs churn so the others stay clean.

2. Hub and spoke, on the same skeleton

Same layout, same box positions, hub and spoke terminology. Read this against the ARA diagram above; every box that changed is a difference between the systems.

Spoke init instead of session record read
A spoke loads the preamble, its status file, its spoke log, and project context verbatim. Same continuity mechanism as ARA, but the handoff lives in the Current Step and Handoff sections of the status file rather than a dedicated session record.
File reads instead of Foresight
There is no inference over memory. The spoke reads text and reasons over whatever it loaded in context. It can recall what is written; it cannot ask the store a question whose answer is not literally recorded, and it gets no confidence labels or falsifiers.
/spoke-update instead of research-manager
Same trigger discipline (checkpoints and session end), same blocking single writer property, same write isolation (a spoke writes only its own status and log). But the output is freeform markdown: a status snapshot overwrite, an appended timestamped log entry, and a capture nudge into the project learnings file. There is no typed routing into layers, no IDs, no claim structure, so the record is a chronological journal rather than a traversable knowledge graph of how the project went.
Project files instead of a layered artifact
The closest correspondences: spoke logs with their mandated dead end format approximate /trace, learnings approximate /logic without the claim schema, and linked docs or sheets approximate /evidence. The bindings between them are prose mentions, not ID references.

What hub and spoke has that ARA does not

ARA is a single project, single thread design. Hub and spoke is an operating system across projects: multiple spoke sessions run in parallel, each an independent session bound to one project or task, and a Base session coordinates through the files.

Base as active hub
Base runs standup, consolidates spoke statuses into the work log at wrap-up, and promotes cross-project learnings into knowledge/. ARA has no equivalent process; its artifact is the hub. Consolidation therefore happens out of band here (a human paced Base session) versus inline in ARA (the maturity tracker at every closure).
Scale model
ARA: one artifact, one project, sessions strictly sequential against it. Hub and spoke: many projects, many concurrent sessions, made safe by write isolation rather than by having a single writer.

3. Differences at a glance

Dimension ARA Hub and spoke
Read path Foresight: inference over memory with confidence and falsifiers Plain file reads; recall only
Recorder output Typed entries routed into layers, IDs, claim schema with falsification Freeform markdown: snapshot, appended log, prose learnings
Store shape Knowledge graph: claims cite trace nodes cite evidence by ID Chronological journal: prose files linked by mention
Refutation Status transition on the claim, correction cites old ID Append a new entry; old text stays but carries no status
Write isolation / concurrency Single writer, sessions strictly sequential Per-spoke writer, many concurrent sessions; Base alone writes cross-project state
Scope One project, one artifact, sequential sessions Many projects, parallel spoke sessions, Base as hub
Consolidation Inline: maturity tracker at every closure Out of band: Base promotes learnings at wrap-up
The one line summary. ARA is deeper: a structured, falsifiable, auditable memory for one project. Hub and spoke is wider: a coordination layer for many projects and parallel sessions, whose memory is prose. The natural synthesis is hub and spoke's shell with an ARA shaped store inside each project: keep spokes, Base, and write isolation; upgrade capture to a claim schema; add a foresight style read skill.

4. The synthesis architecture

Hub and spoke's shell, ARA's store. Two views: first the inside of one spoke, then how multiple spokes stay consistent.

4.1 Inside one spoke

The spoke session keeps its skeleton. The recorder is triggered at closure; its typed writes land in a private workspace. Foresight is a read only call from the work loop, and shared /logic and knowledge/ sit outside the workspace because the spoke never writes them. Seal audits the full artifact.

Harvester, in this setting
Fires at every closure and at session end. It scans everything since the previous checkpoint: the transcript, tool outputs (MAST job results, SQL query outputs, eval numbers), and config diffs. Each extracted event carries a provenance tag (user asserted, agent suggested, agent executed) and a pointer to its raw source, which becomes the /evidence entry. The old freeform status snapshot survives as one harvested event type among others, so nothing about resumability is lost.
/src, in this setting
Experiment configs, launch commands, and slash command definitions, each stored with the rationale for its settings. This is the layer the old system scattered across the queue sheet's config columns; here it is spoke owned and ID addressable, so a trace node can cite exactly the config that produced it.
staging/, in this setting
Per spoke, which is the detail that makes concurrency trivial: half formed beliefs never touch a shared file. The Mature step re-checks staged items at every subsequent closure, where it matures, merges into an existing claim, or dies. The spoke's Mature step only marks a claim as candidate ripe within its own staging; the move into shared /logic is done exclusively by Base at the merge. Spoke sessions can live for days across many merges; whatever has ripened by each evening gets crystallized that evening. Staged beliefs are already visible to every reader through foresight the moment they land, so merge latency delays authority (ID, dedup, promotion eligibility), not knowledge.
Foresight, in this setting
Called at two moments: spoke init (replacing part of the verbatim file load with an actual briefing: current beliefs, open questions, known dead ends for this task) and before proposing any experiment (predicted outcome, confidence, falsifier). Its read scope is deliberately wider than the spoke's write scope: shared /logic and knowledge/ first, then the global claim store, then any spoke's trace. Zero writes is what makes that unrestricted read scope safe under concurrency.

4.2 Tiers, concurrency, and the merge

Storage is arranged in three tiers, distinguished by exactly two properties: who may write, and at which synchronization point writes happen. Reads are unrestricted at every tier; the tiers govern writes only.

The tier boundary is exactly the raw-versus-belief boundary. Everything raw is tier 1 and spoke local (evidence, trace, src, staging), because raw material is inherently the record of one spoke's own search and is complete only at that individual level. Only beliefs are shared (tier 2 project /logic, tier 3 knowledge/), because only beliefs are worth integrating across spokes, and integration means citation, not copying. A claim whose support spans several spokes is a tier 2 record citing trace nodes across their tier 1 workspaces by ID; the global reasoning structure is never materialized as one artifact, it exists latently in that citation graph and foresight traverses it on demand. Trace being incomplete at the spoke level is therefore the correct state, not a defect: completeness lives in the citation graph, not in any single trace file.

Tier Contents Sole writer Writes happen
1 Spoke workspaces: evidence, trace, src, staging The owning spoke Continuously, at every closure
2 Project /logic: crystallized claims with IDs Base At the merge: nightly wrap-up, or on demand /merge
3 knowledge/: cross-project claims Base Promotion step of the same merge
What the merge moves, and what it leaves
The merge is claim only. Base reads tier 1 staging and crystallizes ripe claims into tier 2 /logic; nothing else is copied upward. Trace, evidence, and src never leave the spoke workspace: they are the permanent substrate, and tier 2 claims reach back into them by ID (a claim's Sources field points at spoke evidence, its derivation at spoke trace nodes). This is what makes /logic an index over spoke owned raw material rather than a duplicate of it.
What happens to each layer after a merge
In the spoke workspace: trace, evidence, and src are unchanged; staging is the only layer that moves, shrinking as each item is either crystallized (leaves staging, now lives as a tier 2 claim), absorbed (merged into an existing claim it updated), or dropped (a dead candidate, deleted). In the project workspace: tier 2 /logic grows by exactly the new and revised claims, each a thin record pointing back down into tier 1.
Why this never conflicts
Tier 1 partitions are disjoint, so any number of spokes writes in parallel with no locking. Tiers 2 and 3 have one writer, Base, acting at one serialized moment, the merge. There is no synchronization point at which two writers can touch the same file.
Everything is a spoke. The entire concurrency model reduces to one rule: every session is a spoke that writes only its own workspace. A normal spoke's workspace is its tier 1 partition. Base is a special spoke whose workspace is tiers 2 and 3, and the merge is simply Base's /spoke-update. The Seal is a special spoke whose workspace is its report file. Same init, update, and handoff machinery everywhere; the only thing that varies per session is the ownership declaration.
Three loops, three cadences. Continuous visibility: staging plus foresight, zero latency, any spoke's fresh belief is readable everywhere the moment it lands. Daily canonicity: the merge, nightly with on demand escalation, confers IDs, dedup, and promotion. Weekly rigor: the Seal, deep audit by a fresh agent. Each loop runs exactly as fast as its output is perishable.
