ARA and hub and spoke, side by side

One skeleton, two systems. Compiled July 2026 from the ARA paper (arXiv 2604.24658), the research world model blog post, and the claude-templates skill set.

Both systems make research state outlive the context window: recorded work that a future session can read, trust, and build on. This document defines ARA first, then re-renders the hub and spoke system onto the identical diagram skeleton so that every difference is visible as a changed box rather than a new picture.

1. What ARA is

An ARA (Agent-Native Research Artifact) is a per-project repository that serves as the research memory, plus three skills that operate on it. There is no orchestrator: the host agent (any Claude Code style session) does the work, and coordination happens entirely through the files. One agent, one thread; the recorder blocks the loop briefly while it writes, then work resumes. The Seal is the only asynchronous element, a separate periodic session.

Closure moment
Any point where something resolves: an experiment produces a result, a hypothesis is confirmed or killed, a dead end is hit, or the session ends. Most closures happen mid session. Raw evidence lands in /evidence continuously; only the epistemic layers wait for outcomes.
Foresight
Read only, pulled, never scheduled. Answers questions whose answers are not literally recorded by composing existing claims, labeling grounded steps versus speculative leaps, attaching confidence, and naming what observation would falsify the prediction.
Repository
Four layers. /logic holds claims, concepts, and heuristics as structured entries with IDs. /trace holds the exploration graph including dead ends. /evidence holds raw outputs. /src holds code and configs with rationale. Cross references by ID make the whole thing a traversable knowledge graph of how the project went.
Seal
Periodic audit in three levels: structural integrity, epistemic rigor, execution reproducibility. Failure blocks nothing; findings are written back as work items and the artifact carries no trust stamp until they are addressed.

The research-manager, zoomed in

Every step is the same LLM following a rubric in the skill file. The router decides what type of object each harvested event is; only belief bound events pass through a ripeness gate.

2. Hub and spoke, on the same skeleton

Same layout, same box positions, hub and spoke terminology. Read this against the ARA diagram above; every box that changed is a difference between the systems.

Spoke init instead of session record read
A spoke loads the preamble, its status file, its spoke log, and project context verbatim. Same continuity mechanism as ARA, but the handoff lives in the Current Step and Handoff sections of the status file rather than a dedicated session record.
File reads instead of Foresight
There is no inference over memory. The spoke reads text and reasons over whatever it loaded in context. It can recall what is written; it cannot ask the store a question whose answer is not literally recorded, and it gets no confidence labels or falsifiers.
/spoke-update instead of research-manager
Same trigger discipline (checkpoints and session end), same blocking single writer property, same write isolation (a spoke writes only its own status and log). But the output is freeform markdown: a status snapshot overwrite, an appended timestamped log entry, and a capture nudge into the project learnings file. There is no typed routing into layers, no IDs, no claim structure, so the record is a chronological journal rather than a traversable knowledge graph of how the project went.
Project files instead of a layered artifact
The closest correspondences: spoke logs with their mandated dead end format approximate /trace, learnings approximate /logic without the claim schema, and linked docs or sheets approximate /evidence. The bindings between them are prose mentions, not ID references.

What hub and spoke has that ARA does not

ARA is a single project, single thread design. Hub and spoke is an operating system across projects: multiple spoke sessions run in parallel, each an independent Claude Code session bound to one project or task, and a Base session coordinates through the files.

Base as active hub
Base runs standup, consolidates spoke statuses into the work log at wrap-up, and promotes cross-project learnings into knowledge/. ARA has no equivalent process; its artifact is the hub. Consolidation therefore happens out of band here (a human paced Base session) versus inline in ARA (the maturity tracker at every closure).
Scale model
ARA: one artifact, one project, sessions strictly sequential against it. Hub and spoke: many projects, many concurrent sessions, made safe by write isolation rather than by having a single writer.

The one line summary. ARA is deeper: a structured, falsifiable, auditable memory for one project. Hub and spoke is wider: a coordination layer for many projects and parallel sessions, whose memory is prose. The natural synthesis is hub and spoke's shell with an ARA shaped store inside each project: keep spokes, Base, and write isolation; upgrade capture to a claim schema; add a foresight style read skill.

4. The synthesis architecture

Hub and spoke's shell, ARA's store. Two views: first the inside of one spoke, then how multiple spokes stay consistent.

4.1 Inside one spoke

The spoke session keeps its skeleton. The recorder is the ARA manager verbatim, triggered at /spoke-update checkpoints; its typed writes land in a workspace holding all five store components. Foresight is a read only call from the work loop, and the shared /logic layer sits outside the workspace because the spoke never writes it.

Harvester, in this setting
Fires at every /spoke-update checkpoint and at session end. It scans everything since the previous checkpoint: the transcript, tool outputs (MAST job results, SQL query outputs, eval numbers), and config diffs. Each extracted event carries a provenance tag (user asserted, agent suggested, agent executed) and a pointer to its raw source, which becomes the /evidence entry. The old freeform status snapshot survives as one harvested event type among others, so nothing about resumability is lost.
/src, in this setting
Experiment configs, launch commands, and slash command definitions, each stored with the rationale for its settings. This is the layer the old system scattered across the queue sheet's config columns; here it is spoke owned and ID addressable, so a trace node can cite exactly the config that produced it.
staging/, in this setting
Per spoke, which is the detail that makes concurrency trivial: half formed beliefs never touch a shared file. The Mature step re-checks staged items at every subsequent checkpoint, where they ripen, merge into an existing claim, or die. Nothing is ever promoted out of staging directly; only ripe claims move, and only at wrap-up.
Foresight, in this setting
Called at two moments: spoke init (replacing part of the verbatim file load with an actual briefing: current beliefs, open questions, known dead ends for this task) and before proposing any experiment (predicted outcome, confidence, falsifier). Its read scope is deliberately wider than the spoke's write scope: shared /logic first, then the global claim store, then any spoke's trace. Zero writes is what makes that unrestricted read scope safe under concurrency.

4.2 Concurrency and the merge

The rule is one writer per file, arranged in three tiers, and every cross tier move is executed by exactly one actor, Base, at wrap-up. It is the git model: workspaces are branches, wrap-up is the merge.

Why this never conflicts
During the day, spokes touch only disjoint file sets, so any number can run in parallel with no locking. The two shared tiers each have exactly one writer, Base, which runs at a single moment, wrap-up, when the spokes are idle. There is no window in which two writers can race on the same file. ARA achieved consistency with one sequential writer per artifact; this achieves it with one writer per file plus a serialized merge point.
What Base does at the merge
Reads every workspace's staging and /logic candidates, crystallizes ripe claims into project /logic with IDs and provenance, deduplicates against existing claims (widening scope beats adding a near duplicate), promotes claims that generalize across projects into knowledge/, and runs the light audit while it is already reading everything: do cited evidence IDs resolve, did any recorded falsifier fire against the day's results. Findings become next standup's work items.
Where sealing happens
Split by level, matching cost and independence to cadence. Level 1 (structural: cited IDs resolve, recorded falsifiers checked against new results, entries well formed) runs inline in Base's wrap-up; it is mechanical, cheap, and judgment free, so Base running it on its own merge is safe. Levels 2 and 3 (rigor: do conclusions follow from cited evidence; reproducibility: re-run selected /src configs and confirm the evidence regenerates) run in a separate periodic Seal session, a fresh agent with no stake in the work, slotted into the weekly review cadence. This preserves ARA's key audit property, that the auditor is never the writer being audited. The Seal writes findings only to its own report file, which Base reads at the next standup and turns into work items: demote or re-scope flagged claims, fix broken citations, re-run failed reproductions.
The two-spokes-one-project case
A1 and A2 above share a project but never a file: each has its own trace, evidence, src, and staging. If both stage overlapping beliefs, the conflict surfaces at the merge, where Base sees both candidates side by side and reconciles them into one claim, exactly like a git merge conflict resolved by the integrator rather than by the branch authors.
